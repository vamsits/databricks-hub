Platform Overview

What is Databricks?

Databricks is a unified analytics platform built on Apache Spark, providing a collaborative workspace for data engineering, machine learning, and business analytics — all atop an open Lakehouse architecture.

The Unified Lakehouse Platform

Founded in 2013 by the creators of Apache Spark at UC Berkeley's AMPLab, Databricks has grown into the leading unified data and AI platform. The platform combines the best of data lakes (low-cost storage, flexibility) and data warehouses (ACID transactions, query performance) into a single architecture called the Lakehouse.

Databricks runs on all major clouds (AWS, Azure, GCP) and provides a fully managed Apache Spark environment with enhanced capabilities including Delta Lake, Photon (vectorized query engine), Unity Catalog (data governance), and Mosaic AI (MLOps).

✓
Open-source foundation

Built on Apache Spark, Delta Lake, MLflow — open standards you can trust.

✓

✓
Unified for all personas

Data engineers, data scientists, analysts, and ML engineers on one platform.

✓

Choose serverless SQL/jobs for simplicity or classic clusters for control.

The Lakehouse Architecture

The Lakehouse paradigm emerged to solve the fundamental tension between data lakes (cheap, flexible, but unstructured) and data warehouses (fast, governed, but expensive and rigid).

❌ Data Lake Only

  • ⚠ No ACID transactions
  • ⚠ Schema on read (fragile)
  • ⚠ Poor query performance
  • ⚠ No data versioning
  • ⚠ Difficult governance

⚠ Data Warehouse Only

  • ⚠ Expensive at scale
  • ⚠ Rigid schema
  • ⚠ Proprietary formats
  • ⚠ No ML/AI workloads
  • ⚠ Data duplication

✅ Lakehouse (Databricks)

  • ✓ ACID on cheap object storage
  • ✓ Schema enforcement + evolution
  • ✓ Photon-accelerated queries
  • ✓ Time travel & audit logs
  • ✓ Single governance plane
💡
Key Insight

The Lakehouse uses Delta Lake (open-source) as a metadata/transaction layer on top of Parquet files in object storage (S3/ADLS/GCS). You keep data costs low while gaining warehouse-grade reliability.

Delta Lake

Delta Lake is an open-source storage layer that brings ACID transactions, scalable metadata handling, and data versioning to Apache Spark and big data workloads. It's the storage foundation of the Lakehouse.

Core Capabilities

✓
ACID Transactions

Full atomicity, consistency, isolation, and durability. Supports concurrent readers and writers without corruption.

✓
Time Travel

Query any historical version of your table. Rollback data mistakes with RESTORE TABLE. Audit changes over time.

✓
Schema Enforcement & Evolution

Reject bad data automatically. Safely evolve schemas with mergeSchema or explicit DDL.

✓
Liquid Clustering (DBR 13.3+)

Replaces static partitioning. Automatically re-clusters data for optimal query performance without partition explosion.

✓
Change Data Feed (CDF)

Track row-level changes (insert/update/delete) for efficient downstream incremental processing.

Delta Lake SQL Examples

-- Create a Delta table with optimized layout
CREATE TABLE sales.transactions (
  transaction_id BIGINT,
  customer_id    INT,
  amount         DECIMAL(18,2),
  ts             TIMESTAMP
)
USING DELTA
CLUSTER BY (customer_id, ts)  -- Liquid Clustering
TBLPROPERTIES (
  'delta.enableChangeDataFeed' = 'true',
  'delta.columnMapping.mode'  = 'name'
);

-- Time travel query
SELECT * FROM sales.transactions
  VERSION AS OF 42;

-- Restore to previous version
RESTORE TABLE sales.transactions
  TO VERSION AS OF 41;

-- Optimize with Z-ordering (legacy)
OPTIMIZE sales.transactions
  ZORDER BY (customer_id, ts);

-- Vacuum stale files (default 7-day retention)
VACUUM sales.transactions
  RETAIN 168 HOURS;
💡
Liquid Clustering vs Z-Order

Liquid Clustering (DBR 13.3+) is the successor to Z-ORDER. It's incremental, doesn't require full rewrites, and automatically adapts to query patterns. Prefer it for new tables.

Photon Engine

Photon is Databricks' vectorized query engine written in C++. It accelerates SQL and DataFrame operations by processing data in column batches using SIMD (Single Instruction, Multiple Data) CPU instructions, dramatically reducing query latency and CPU costs.

Performance Benchmarks

Aggregations8x faster
Joins (hash)12x faster
Scans (wide tables)6x faster
Sort operations4x faster
⚠️
Photon Limitations

Photon doesn't accelerate Python UDFs, pandas UDFs (use Arrow-based UDFs instead), or arbitrary RDD operations. Keep transformations in native Spark SQL / DataFrames for maximum benefit.

Enabling & Optimizing Photon

# Photon is enabled by default on DBR 9.1+
# Check if Photon is active in your cluster
spark.conf.get("spark.databricks.photon.enabled")
# Returns: 'true'

# Force Photon for specific operations
spark.conf.set(
  "spark.databricks.photon.allDataSources.enabled",
  "true"
)

# Photon-friendly: native SQL aggregations
spark.sql("""
  SELECT 
    customer_id,
    SUM(amount) AS total,
    COUNT(*) AS cnt,
    AVG(amount) AS avg_amount
  FROM sales.transactions
  WHERE ts >= '2024-01-01'
  GROUP BY customer_id
  ORDER BY total DESC
  LIMIT 100
""")

# Arrow-optimized pandas UDF (works with Photon)
from pyspark.sql.functions import pandas_udf
import pandas as pd

@pandas_udf("double")
def compute_score(s: pd.Series) -> pd.Series:
    return s * 1.15 + 100
💡
Cost Impact

Photon can reduce job runtime significantly, but Photon-enabled clusters cost 2x DBU rate. Net effect: typically 30–60% cost savings due to faster completion, especially for SQL-heavy workloads.

Unity Catalog

Unity Catalog is Databricks' unified governance layer that provides centralized access control, auditing, lineage tracking, and data discovery across all workspaces and clouds in your organization.

Three-Level Namespace

-- Unity Catalog 3-level namespace:
-- catalog.schema.table

-- Create catalog
CREATE CATALOG production;

-- Create schema (database)
CREATE SCHEMA production.sales
  COMMENT 'Sales domain tables';

-- Grant privileges
GRANT USE CATALOG ON CATALOG production
  TO `data-analysts`;

GRANT SELECT ON SCHEMA production.sales
  TO `data-analysts`;

-- Column-level masking
ALTER TABLE production.sales.customers
  ALTER COLUMN email
  SET MASK catalog.security.email_mask;

-- Row-level security
CREATE ROW ACCESS POLICY region_filter
AS (region STRING) RETURN
  is_account_group_member(region)
  OR is_account_group_member('admins');

Delta Live Tables (DLT) Example

import dlt
from pyspark.sql.functions import col, expr

# Bronze layer — raw ingestion
@dlt.table(
  comment="Raw events from Kafka",
  table_properties={"quality": "bronze"}
)
def raw_events():
  return (
    spark.readStream
      .format("kafka")
      .option("subscribe", "events")
      .load()
  )

# Silver layer — with expectations
@dlt.table()
@dlt.expect_or_drop("valid_id", "id IS NOT NULL")
@dlt.expect_or_drop("valid_ts", "ts > '2020-01-01'")
def cleaned_events():
  return (
    dlt.read_stream("raw_events")
      .select(
        col("value").cast("string").alias("payload"),
        col("timestamp").alias("ts")
      )
  )

# Gold layer — business aggregation
@dlt.table(
  comment="Hourly event counts by type"
)
def hourly_metrics():
  return (
    dlt.read("cleaned_events")
      .groupBy(expr("DATE_TRUNC('hour', ts)").alias("hour"))
      .count()
  )

Architecture Deep Dive

Understanding the control plane, data plane, cluster internals, and deployment topology.

Control Plane vs. Data Plane

Databricks uses a split-plane architecture. The control plane is managed by Databricks and hosts the web UI, cluster management, job scheduler, and APIs. The data plane runs in your cloud account — all compute and data stays with you.

🔐
Security Implication

Your data never leaves your cloud account. The control plane only receives metadata (cluster status, job results) — never your actual data. This is critical for compliance (HIPAA, GDPR, FedRAMP).

Control Plane Components

✓
Workspace UI

Notebooks, jobs, SQL editor, dashboards, model registry, data explorer.

✓
Cluster Manager

Provision, configure, and terminate clusters. Autoscaling decisions made here.

✓
Jobs Scheduler

Trigger jobs on schedule, event-driven, or via REST API. DAG execution engine.

✓
Unity Catalog Metastore

Centralized metadata store. One per region, shared across all workspaces.

Spot Spot & Preemptible Instances

Using spot/preemptible instances is one of the highest-impact cost reduction levers — typically 60–90% cheaper than on-demand. The tradeoff is potential interruption, which Databricks handles gracefully with automatic retries and fallback.

# AWS: SPOT_WITH_FALLBACK (recommended for production)
{
  "aws_attributes": {
    "availability": "SPOT_WITH_FALLBACK",
    "spot_bid_price_percent": 100,  # Bid at on-demand price (max availability)
    "first_on_demand": 1            # Keep driver on-demand for stability
  }
}

# Azure: Use Spot VMs with eviction policy
{
  "azure_attributes": {
    "availability": "SPOT_WITH_FALLBACK_AZURE",
    "spot_bid_max_price": -1,  # -1 = current on-demand price
    "first_on_demand": 1
  }
}

# GCP: Use preemptible VMs
{
  "gcp_attributes": {
    "use_preemptible_executors": true,
    "availability": "PREEMPTIBLE_WITH_FALLBACK_GCP"
  }
}

# Making jobs spot-resilient: checkpointing
spark.conf.set(
  "spark.databricks.streaming.statefulOperator.asyncCheckpointing.enabled",
  "true"
)
# Enable shuffle service for spot-safe job recovery
spark.conf.set("spark.databricks.shuffleService.enabled", "true")

Sizing Cluster Rightsizing

Over-provisioned clusters are one of the top sources of wasted DBU spend. Use Ganglia metrics, Spark UI, and system tables to right-size your clusters.

Rightsizing Framework

-- Query system tables for cluster utilization
SELECT
  cluster_id,
  cluster_name,
  AVG(driver_memory_utilization) AS avg_driver_mem,
  AVG(worker_memory_utilization) AS avg_worker_mem,
  AVG(worker_cpu_utilization)    AS avg_cpu,
  MAX(num_workers)               AS max_workers_used
FROM system.compute.cluster_events
WHERE timestamp > DATEADD(day, -30, NOW())
GROUP BY 1, 2
HAVING avg_cpu < 0.4  -- <40% CPU = oversized
ORDER BY avg_cpu;

Worker Count Sizing Guide

# Quick formula for ETL jobs:
# workers = (total_data_GB / per_worker_cache_GB) * 1.5

## Example sizing calculation
total_data  = 500  # GB of compressed Parquet
target_time = 30   # minutes
node_type   = "r5.4xlarge"  # 128GB RAM, 16 vCPU

# Spark partitions rule of thumb
# target: 100–200MB per partition after decompression
decompressed_gb   = total_data * 3   # ~3x expansion
target_part_mb    = 128
partitions_needed = (decompressed_gb * 1024) / target_part_mb
# = 500 * 3 * 1024 / 128 = ~12000 partitions

# Workers needed (16 vCPU per worker)
# Aim for 2–4 tasks per vCPU
tasks_per_vcpu = 3
workers        = partitions_needed / (tasks_per_vcpu * 16)
# = 12000 / 48 = ~250 workers (aggressive)
# Realistically: 20–50 workers + autoscaling

# Override partitions at runtime:
spark.conf.set("spark.sql.shuffle.partitions", 2000)

Storage Storage Cost Optimization

Storage costs are often underestimated. Between table sprawl, unvacuumed Delta tables, and suboptimal file layouts, storage can easily 3–5x its necessary cost.

Delta Table Storage Hygiene

-- 1. Check table storage consumption
DESCRIBE DETAIL prod.sales.transactions;
-- Shows: numFiles, sizeInBytes, numDeletedRecords

-- 2. VACUUM to remove old file versions
-- Default: retains 7 days (for time travel)
SET spark.databricks.delta.retentionDurationCheck.enabled = false;
VACUUM prod.sales.transactions RETAIN 72 HOURS;

-- 3. OPTIMIZE to compact small files
-- Target: 256MB–1GB file size
OPTIMIZE prod.sales.transactions;

-- 4. Auto-optimize settings
ALTER TABLE prod.sales.transactions
  SET TBLPROPERTIES (
    'delta.autoOptimize.optimizeWrite' = 'true',
    'delta.autoOptimize.autoCompact'   = 'true',
    'delta.targetFileSize'             = '268435456'  -- 256MB
  );

-- 5. Convert managed to external for cost control

from delta.tables import DeltaTable

delta_table = DeltaTable.forName(spark, "silver.customers")

(delta_table.alias("t")
  .merge(
    source=updates_df.alias("u"),
    condition="t.customer_id = u.customer_id"
  )
  .whenMatchedUpdate(set={
    "name": "u.name",
    "updated_at": "u.updated_at"
  })
  .whenNotMatchedInsert(values={
    "customer_id": "u.customer_id",
    "name": "u.name",
    "updated_at": "u.updated_at"
  })
  .execute()
)

# 2. Optimize MERGE performance:
# a) Filter target table partition first
MERGE INTO silver.customers t
USING (
  SELECT * FROM staging.updates
  WHERE region = 'us-west'
) u
ON t.customer_id = u.customer_id
AND t.region = 'us-west'   -- partition filter on target!

Structured Streaming Performance

from pyspark.sql import SparkSession
from pyspark.sql.functions import from_json, col, window
from pyspark.sql.types import *

# Kafka → Delta streaming pipeline
schema = StructType([
  StructField("user_id", LongType()),
  StructField("event_type", StringType()),
  StructField("amount", DoubleType()),
  StructField("ts", TimestampType())
])

stream_df = (spark
  .readStream
  .format("kafka")
  .option("kafka.bootstrap.servers", "broker:9092")
  .option("subscribe", "transactions")
  .option("startingOffsets", "latest")
  .option("maxOffsetsPerTrigger", 100000)  # rate limiting
  .load()
  .select(from_json(col("value").cast("string"), schema).alias("data"))
  .select("data.*")
)

# Windowed aggregation with watermark
agg_df = (stream_df
  .withWatermark("ts", "10 minutes")   # Late data tolerance
  .groupBy(
    window(col("ts"), "5 minutes"),
    col("event_type")
  )
  .sum("amount")
)

# Write to Delta with optimized settings
query = (agg_df
  .writeStream
  .outputMode("append")             # Use append for watermarked aggs
  .format("delta")
  .option("checkpointLocation", "/Volumes/prod/checkpoints/txn_agg")
  .option("mergeSchema", "false")
  .trigger(processingTime="30 seconds")  # or once=True for micro-batch
  .toTable("gold.transaction_aggregates")
)

Streaming Tuning Checklist

✓
Set maxOffsetsPerTrigger

Prevents large catch-up batches from OOM-ing the driver. Start at 100K–1M per Kafka topic.

✓
Use availableNow trigger

trigger(availableNow=True) processes all available data then stops — perfect for scheduled streaming jobs.

✓
Set watermark before aggregation

Without watermark, stateful operations retain state forever → OOM. Watermark bounds state size.

✓
Async checkpointing

Enable asyncCheckpointing to overlap checkpoint I/O with processing, improving throughput.

✓
Target partition size: 128–256MB

Tune spark.databricks.streaming.target.batchSizeInBytes for optimal Delta writes.

Adaptive Query Execution (AQE)

AQE (enabled by default in DBR 8+) re-optimizes query plans at runtime based on actual data statistics gathered during execution. It's one of the most impactful Spark features for production workloads.

AQE Features

After a shuffle, AQE merges small output partitions into fewer, larger ones. This reduces task scheduling overhead and improves executor efficiency.

spark.sql.adaptive.coalescePartitions.enabled = true
spark.sql.adaptive.coalescePartitions.minPartitionSize = 64MB
spark.sql.adaptive.advisoryPartitionSizeInBytes = 256MB
# Result: 10,000 tiny partitions → 200 healthy ones

AQE can switch from sort-merge join to broadcast join at runtime if it discovers one side is small enough — without you manually setting hints.

spark.sql.adaptive.enabled = true
# AQE will auto-broadcast if runtime size < threshold
spark.sql.autoBroadcastJoinThreshold = 50m
# Monitor with: df.explain("formatted")

AQE detects skewed partitions and splits them into sub-partitions, then replicates the corresponding rows from the non-skewed side — eliminating the long-tail stragglers.

spark.sql.adaptive.skewJoin.enabled = true
spark.sql.adaptive.skewJoin.skewFactor = 5
# A partition is skewed if:
# size > skewFactor * median AND size > 256MB

Predictive I/O

Predictive I/O is a Databricks-exclusive enhancement (Serverless, DBR 12.2+) that uses ML to predict which columns and row groups to prefetch based on query patterns.

✓
Automatic — no configuration needed

Enabled by default on Serverless SQL Warehouses. No knobs to turn.

✓
Learning per-table patterns

Analyzes historical query patterns to prefetch likely-needed data into the buffer pool.

✓
Up to 3x I/O reduction

Particularly effective for dashboards with repeated filtering on the same columns.

Databricks Glossary

Every key term explained — from DBU to Liquid Clustering.

DBU (Databricks Unit)

The billing unit for Databricks compute. 1 DBU = 1 unit of processing capacity per hour. Price per DBU varies by product (Jobs, All-Purpose, SQL), cloud provider, and tier (Standard vs. Premium). Photon doubles the DBU rate but reduces runtime.

Billing

Delta Lake

Open-source storage layer providing ACID transactions, scalable metadata, schema enforcement, time travel, and change data feed on top of Parquet files in object storage. The foundation of the Databricks Lakehouse.

Storage

Unity Catalog

Databricks' unified data governance solution. Provides centralized access control, auditing, lineage tracking, and data discovery across all workspaces and clouds. Uses a 3-level namespace: catalog.schema.table.

Governance

Photon Engine

Databricks' vectorized query execution engine written in C++. Uses SIMD instructions to process columnar data in batches. Significantly accelerates SQL and DataFrame operations. Enabled by default on DBR 9.1+ with 2x DBU cost.

Performance

DLT (Delta Live Tables)

Declarative ETL framework for building reliable data pipelines. Define transformations as SQL or Python; DLT handles dependencies, data quality expectations, auto-scaling, and monitoring automatically.

Pipelines

Liquid Clustering

Successor to static partitioning and Z-ordering in Delta Lake (DBR 13.3+). Incrementally reorganizes data layout based on clustering keys without full rewrites. Adapts automatically to changing query patterns.

Mosaic AI

Databricks' end-to-end AI/ML platform. Includes MLflow (experiment tracking, model registry), Feature Store, Model Serving, Vector Search, AI Playground, and the Agent Framework for building compound AI systems.

AI/ML

AQE (Adaptive Query Execution)

Spark 3.0+ feature that re-optimizes query plans at runtime using actual data statistics. Key features: dynamic partition coalescing, dynamic join strategy switching, and skew join optimization. Enabled by default in DBR 8+.

Spark

CDF (Change Data Feed)

Delta Lake feature that records row-level changes (inserts, updates, deletes) as a separate change log. Enables efficient downstream CDC processing and reduces the need for expensive full table scans.

Delta Lake

DBFS (Databricks File System)

Distributed file system abstraction layer on top of object storage. Being deprecated in favor of Unity Catalog Volumes for new workloads. Still supported but avoid DBFS root for new code.

Storage

Auto Loader

Incremental file ingestion tool (cloudFiles format) that efficiently processes new files as they arrive in cloud storage. Supports file notification mode for extremely high file volumes with minimal listing overhead.

Ingestion

DABs (Databricks Asset Bundles)

Infrastructure-as-Code framework for managing all Databricks resources (jobs, clusters, notebooks, pipelines, models) in version-controlled YAML. The recommended CI/CD approach replacing dbx and Databricks Repos-based deployments.

DevOps

Deletion Vectors

Delta Lake feature (DBR 12.2+) that enables soft-deletes — marking rows as deleted in a side-file without rewriting data files. Dramatically improves DELETE and UPDATE performance, especially for GDPR compliance.

Delta Lake

Delta Sharing

Open protocol for securely sharing live data across organizations, clouds, and platforms without copying it. Recipients can access shared data via Databricks, pandas, Apache Spark, or any Delta Sharing client.

Sharing

Spot / Preemptible Instances

Cloud provider unused capacity available at 60–90% discount vs. on-demand pricing. Can be interrupted with short notice. Databricks handles interruptions gracefully with SPOT_WITH_FALLBACK mode and shuffle service.

Cost

Z-Order

Multi-dimensional data clustering technique in Delta Lake that co-locates related data in the same files to improve data skipping. Used with OPTIMIZE ZORDER BY (col1, col2). Being superseded by Liquid Clustering.
